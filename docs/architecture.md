MoltPulse follows a modular, domain-driven architecture that separates concerns and enables extensibility.

### 1. Collect Phase
Collectors run in parallel using `ThreadPoolExecutor`. Each collector:
- Checks if required API keys are available
- Fetches data from external APIs or RSS feeds
- Returns normalized items in a common schema

### 2. Process Phase
Processing applies to all collected items:
- **Filter**: Remove items outside date range
- **Score**: Calculate relevance using weighted formula (45% relevance, 25% recency, 30% engagement)
- **Sort**: Order by score descending
- **Dedupe**: Remove duplicate items by URL/content hash

### 3. Generate Phase
Report generators create structured output:
- Domain-specific formatting
- Section organization
- Source citation collection

### 4. LLM Enhance Phase (Optional)
When enabled, reports are enhanced with AI-generated content:
- **Executive Summary**: 2-3 sentence overview of key developments
- **Section Insights**: Per-section analysis (financial, news, PE activity)
- **Strategic Recommendations**: Actionable next steps

LLM backend detection (in priority order):
1. OpenClaw gateway (`OPENCLAW_GATEWAY_URL` or `openclaw gateway probe`)
2. Claude Code CLI (`claude --print` via subprocess)
3. Anthropic SDK (if `ANTHROPIC_API_KEY` set)
4. Disabled (no enhancement)

Custom prompts can be configured per profile (see [Profiles](profiles.md#llm-configuration)).

### 5. Deliver Phase
Delivery sends the report via configured channel:
- **Console**: Print to stdout (default)
- **File**: Save to disk (JSON, HTML, Markdown)
- **Email**: Send via SMTP

## Versioning

MoltPulse uses git-based versioning via `hatch-vcs`:
- Tagged releases: `0.1.1`, `1.0.0`
- Development versions: `0.1.1.dev0+10.g2214efdf3` (commit count + hash)

The version is auto-generated at build time into `_version.py`.

## Collector Architecture

Collectors self-declare their API key requirements and priority:

```python
class AlphaVantageCollector(FinancialCollector):
    REQUIRED_API_KEYS = ["ALPHA_VANTAGE_API_KEY"]
    COLLECTOR_PRIORITY = 0  # Default priority

class NewsDataCollector(NewsCollector):
    REQUIRED_API_KEYS = ["NEWSDATA_API_KEY"]
    COLLECTOR_PRIORITY = 10  # Higher = preferred over NewsAPICollector

class RSSCollector(Collector):
    REQUIRED_API_KEYS = []  # No API key needed

class PEActivityCollector(Collector):
    REQUIRED_API_KEYS = ["INTELLIZENCE_API_KEY", "NEWSDATA_API_KEY"]
    REQUIRES_ANY_KEY = True  # Only one needed
```

**Collector Selection**: When multiple collectors of the same type exist, the orchestrator:
1. Sorts by `COLLECTOR_PRIORITY` (higher first)
2. Prefers collectors with API keys configured
3. Falls back to first available

The orchestrator runs a preflight check to identify available collectors before execution.

### Rate Limiting

Some APIs have strict rate limits. Collectors handle this automatically:

| API | Rate Limit | MoltPulse Handling |
|-----|------------|-------------------|
| Alpha Vantage (free) | 1 req/sec, 25/day | 1.2s delay between requests, reduced depth limits |
| NewsData.io | 200 req/day | Batched queries |
| xAI | Variable | Exponential backoff |

## Scoring Algorithm

Items are scored using a weighted formula:

```
Score = (Relevance × 0.45) + (Recency × 0.25) + (Engagement × 0.30)
```

- **Relevance**: How well the item matches profile keywords and focus entities
- **Recency**: How recent the item is (exponential decay over 30 days)
- **Engagement**: Platform-specific metrics (likes, comments, shares)

## Extension Points

1. **New Collectors**: Implement `Collector` base class
2. **New Domains**: Create domain directory with `domain.yaml`
3. **New Reports**: Implement `ReportGenerator` base class
4. **New Delivery Channels**: Implement `Deliverer` base class
